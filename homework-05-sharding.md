## Цель работы

Реализовать учебный прототип **распределённого in-memory key-value хранилища с шардированием**.

В предыдущих домашних заданиях (ДЗ №2–4) вы решали задачу **репликации** — копирования одних и тех же данных на несколько узлов для отказоустойчивости.
Репликация не решает проблему масштабирования: если данных больше, чем помещается на один узел, все реплики одинаково переполнятся.

**Шардирование** решает эту проблему: данные **разбиваются на непересекающиеся подмножества** (шарды), и каждый узел хранит только свою часть.
Это позволяет масштабировать и объём хранения, и пропускную способность записи.

Ключевой вопрос шардирования — **функция распределения**: как определить, на каком узле живёт конкретный ключ.
Разные функции дают разные trade-off по равномерности распределения, стоимости ребалансировки и поддержке range-запросов.

В этом задании вы реализуете **три стратегии шардирования**, сравните их на практике и увидите,
почему консистентное хэширование с виртуальными узлами стало стандартом индустрии.

## 1. Архитектура

В системе должны быть **три типа процессов**:

1. **ShardNode** — узел хранилища. Каждый узел хранит часть ключей in-memory. Узлы не знают друг о друге напрямую — вся маршрутизация идёт через Router.
2. **Router** — прокси-маршрутизатор. Принимает клиентские запросы, определяет нужный шард по ключу и стратегии, перенаправляет запрос на нужный узел, собирает ответ.
Router поддерживает актуальную карту кластера и знает, какие узлы живы.
3. **CLI / Cluster Controller** — консольное приложение для управления кластером и отправки пользовательских запросов.

Все компоненты общаются по **TCP-сокетам**. Формат сообщений — **JSON Lines** (один JSON-объект на строку, завершается `\n`).

### Что НЕ требуется
- Репликация данных между шардами (данные хранятся в одном экземпляре).
- Хранение на диске — всё только **in-memory**.

## 2. Модель данных

Хранилище на каждом узле: `Map<String, String>`.

Операции:
- `PUT key value` — записать (создать/перезаписать).
- `GET key` — прочитать.
- `DELETE key` — удалить.
- `GET_KEYS` — вернуть список всех ключей на узле (для отладки и миграции).
- `DUMP` — вернуть полное локальное состояние узла.

Ограничения:
- `key` — непустая строка без пробелов.
- `value` — строка.

## 3. Стратегии шардирования

Система должна поддерживать **три стратегии**, переключаемые через CLI.

### 3.1. Range-based sharding

Ключевое пространство разбивается на **непрерывные диапазоны** по лексикографическому порядку ключей.

Router хранит карту диапазонов:
```text
[minKey..boundary1) → Node A
[boundary1..boundary2) → Node B
[boundary2..maxKey] → Node C
```

Границы диапазонов задаются **вручную** через CLI-команду `setRanges`.

**Формат команды:**
```text
setRanges <boundary1> <boundary2> ...
```
Пример: `setRanges f n` для трёх узлов создаёт диапазоны `[""..f)`, `[f..n)`, `[n..+∞)`.

Количество границ = количество узлов − 1. Если количество границ не совпадает — ошибка.

При добавлении/удалении узла границы **не пересчитываются автоматически** — нужно вызвать `setRanges` заново и выполнить ручную миграцию через `migrateData`.
Конкретная реализация команды `migrateData` остается за студентом. Однако выбранный способ не должен нарушать никакие инварианты и логику шардинга.

### 3.2. Hash-mod-N sharding

Номер шарда определяется по формуле:
```text
shardIndex = hash(key) mod N
```
где `N` — текущее количество живых узлов.

Хэш-функция — **SHA-256**, берутся первые 8 байт и преобразуются в `long`. Использовать `Object.hashCode()` **запрещено** (недетерминирован между JVM).

**Критическая проблема hash-mod-N:** при изменении `N` (добавление/удаление узла) почти все ключи меняют принадлежность.
Перемещается ≈ (N−1)/N данных. Это нужно **реализовать** и **продемонстрировать**.

При изменении числа узлов Router должен:
1. Запросить все ключи с каждого узла (`GET_KEYS`).
2. Для каждого ключа пересчитать принадлежность по новому `N`.
3. Переместить данные, чьи шарды изменились (`GET` со старого → `PUT` на новый → `DELETE` со старого).
4. Вывести статистику: сколько ключей переместилось из общего числа.

### 3.3. Consistent hashing с виртуальными узлами

Реализация хэш-кольца:

1. **Кольцо** — `TreeMap<Long, PhysicalNodeId>`. Ключи — позиции на кольце (long), значения — идентификаторы физических узлов.
2. **Виртуальные узлы (vnodes):** каждый физический узел размещается на кольце `V` раз.
Позиция i-го виртуального узла: `hash(nodeId + "#" + i)`. Количество vnodes `V` задаётся через CLI.
3. **Подбор ноды в зависимости от результата хэша:** `hash(key)` → `nodeId ближайшего сервера вперед ПО ЧАСОВОЙ СТРЕЛКЕ`.
В конце рэнжа нужно реализовать "закольцовывание", которое маппит значения, близкие к концу рэнжа -- на первую ноду.
4. **Хэш-функция:** та же SHA-256 → long, что и в hash-mod-N.

Параметр `V` (число vnodes на узел) должен настраиваться через CLI командой `setVnodes <count>`.
По умолчанию `V = 3`. Изменение `V` требует пересчёта кольца и миграции.

**Миграция при добавлении узла:**
1. Новый узел размещается на кольце (V точек).
2. Router определяет, какие ключи на соседних узлах теперь принадлежат новому узлу.
3. Эти ключи переносятся на новый узел.
4. Выводится статистика миграции.

**Миграция при удалении узла:**
1. Все ключи удалённого узла перераспределяются по оставшимся (clockwise successors).
2. Выводится статистика.

**Обратная мапа:** для эффективного удаления нужна `Map<PhysicalNodeId, List<Long>>` — позиции на кольце для каждого физического узла.

## 4. Сетевой протокол

### 4.1. Типы сообщений

**Клиентские запросы (CLI → Router):**
```json
{"requestId":"uuid","type":"CLIENT_PUT","key":"user:1","value":"Alice"}
{"requestId":"uuid","type":"CLIENT_GET","key":"user:1"}
{"requestId":"uuid","type":"CLIENT_DELETE","key":"user:1"}
```

**Router → ShardNode:**
```json
{"requestId":"uuid","type":"SHARD_PUT","key":"user:1","value":"Alice"}
{"requestId":"uuid","type":"SHARD_GET","key":"user:1"}
{"requestId":"uuid","type":"SHARD_DELETE","key":"user:1"}
{"requestId":"uuid","type":"SHARD_GET_KEYS"}
{"requestId":"uuid","type":"SHARD_DUMP"}
```

**Ответы ShardNode → Router:**
```json
{"requestId":"uuid","status":"OK","value":"Alice"}
{"requestId":"uuid","status":"ERROR","errorCode":"KEY_NOT_FOUND"}
```

**Управляющие (CLI → Router):**
```json
{"type":"ADD_NODE","nodeId":"A","host":"localhost","port":9001}
{"type":"REMOVE_NODE","nodeId":"A"}
{"type":"SET_STRATEGY","strategy":"consistent_hashing"}
{"type":"SET_VNODES","count":150}
{"type":"SET_RANGES","boundaries":["f","n"]}
{"type":"MIGRATE_DATA"}
{"type":"CLUSTER_INFO"}
```

### 4.2. Ответы и ошибки

Обязательные коды ошибок:
- `KEY_NOT_FOUND` — ключ не найден.
- `NO_AVAILABLE_NODES` — нет живых узлов.
- `INVALID_CONFIG` — невалидная конфигурация (неверное число границ, vnodes < 1 и т.д.).
- `BAD_REQUEST` — некорректный запрос.
- `NODE_UNAVAILABLE` — целевой узел недоступен.

## 5. CLI-команды

### 5.1. Управление кластером
- `addNode <nodeId> <host> <port>` — добавить узел. При стратегии `consistent_hashing` автоматически запускает миграцию ключей на новый узел.
- `removeNode <nodeId>` — удалить узел. При `consistent_hashing` автоматически мигрирует данные с удаляемого узла. При `hash_mod_n` запускает полную ребалансировку.
- `listNodes` — показать состав кластера.
- `setStrategy range|hash_mod_n|consistent_hashing` — переключить стратегию шардирования. **Допустимо только при пустом хранилище** (или с принудительным `--force`, при котором данные теряются).

### 5.2. Настройка стратегий
- `setRanges <boundary1> <boundary2> ...` — задать границы для range-based.
- `setVnodes <count>` — задать число виртуальных узлов на физический узел. Вызывает пересчёт кольца и миграцию.

### 5.3. Миграция
- `migrateData` — ручной запуск миграции для `range` и `hash_mod_n`. Для `consistent_hashing` миграция автоматическая при `addNode`/`removeNode`.

### 5.4. Пользовательские команды
- `put <key> <value>` — записать значение.
- `get <key>` — прочитать.
- `delete <key>` — удалить.

### 5.5. Отладочные команды
- `dump [--target <nodeId>]` — состояние одного узла.
- `clusterDump` — состояние всех узлов (key count, содержимое).
- `stats` — статистика распределения ключей по узлам: количество ключей на каждом узле, stddev/mean, коэффициент неравномерности (max/avg).
- `ringInfo` — (только для `consistent_hashing`) вывод позиций узлов на кольце.

---

## 6. Health-check

Router должен периодически (каждые 5 секунд) проверять доступность узлов через TCP-соединение.

- Если узел недоступен — Router исключает его из маршрутизации.
- Запросы к ключам недоступного узла возвращают `NODE_UNAVAILABLE`.
- Если узел снова стал доступен — Router возвращает его в маршрутизацию.

## 7. Обязательные демонстрационные сценарии

В репозитории должен быть файл `demo-scenario.md` с пошаговыми командами CLI для каждого сценария.

### Сценарий 1 — Range-based: неравномерность при плохих границах
1. Запустить 3 узла, установить стратегию `range`.
2. Задать границы `setRanges m z` (или другие, создающие неравномерное распределение).
3. Загрузить 100+ ключей через скрипт (например, `user:001`..`user:100`).
4. Показать через `stats`, что распределение неравномерное.
5. Перезадать границы более удачно, выполнить `migrateData`.
6. Показать, что распределение выровнялось.

### Сценарий 2 — Hash-mod-N: катастрофическая ребалансировка
1. Запустить 3 узла, установить стратегию `hash_mod_n`.
2. Загрузить 1000 ключей.
3. Показать `stats` — распределение равномерное.
4. Добавить 4-й узел (`addNode`).
5. Запустить `migrateData`.
6. Показать, что переместилось **≈75% ключей** (вывести точный процент).
7. Показать, что после миграции все `GET` по-прежнему работают корректно.

### Сценарий 3 — Consistent hashing: минимальная миграция
1. Запустить 3 узла, стратегия `consistent_hashing`, `V=150`.
2. Загрузить 1000 ключей.
3. Показать `stats` — распределение относительно равномерное.
4. Добавить 4-й узел.
5. Показать, что переместилось **≈25% ключей** (≈ 1/N).
6. Показать, что все `GET` корректны.

### Сценарий 4 — Влияние числа виртуальных узлов на равномерность
1. Запустить 5 узлов, стратегия `consistent_hashing`.
2. Загрузить 10000 ключей при `V=1` → показать `stats` (неравномерно).
3. Переключить на `V=5` → `stats` (лучше).
4. Переключить на `V=150` → `stats` (почти идеально).
5. Объяснить закономерность.

### Сценарий 5 — Отказоустойчивость: падение узла
1. Стратегия `consistent_hashing`, 3 узла, 1000 ключей.
2. Остановить один узел.
3. Показать, что `GET` к ключам этого узла возвращает `NODE_UNAVAILABLE`.
4. Показать, что `GET`/`PUT` к ключам на живых узлах работает нормально.
5. Поднять узел обратно — показать, что он снова доступен (но данные потеряны, т.к. in-memory).

## 8. Бенчмарки и отчёт

### 8.1. Что измерять
- **Throughput** (`ops/sec`) — для PUT и GET.
- **Latency**: `avg`, `p50`, `p95`, `p99`.
- **Статистика распределения**: `stddev/mean` по ключам на узлах.
- **Процент мигрированных ключей** при добавлении узла.

Результаты сохранить в `benchmarks/results.csv` или `benchmarks/results.json`.

### 8.2. Обязательные прогоны

#### Часть A — Сравнение стратегий по throughput (6 прогонов)
Кластер: **5 узлов**, 10000 ключей, `threads = 16`.
- стратегии: `range`, `hash_mod_n`, `consistent_hashing` (V=150)
- workload: `putRatio = 0.8` (write-heavy) и `putRatio = 0.2` (read-heavy)

Итого: 3 × 2 = 6 прогонов.

#### Часть B — Стоимость ребалансировки (3 прогона)
Кластер: **4 узла**, загружено **10000 ключей**. Добавляется 5-й узел.
- стратегии: `range` (с ручной миграцией), `hash_mod_n`, `consistent_hashing` (V=150)
- измерить: % мигрированных ключей, время миграции

Итого: 3 прогона.

#### Часть C — Влияние vnodes на равномерность (4 прогона)
Кластер: **5 узлов**, **10000 ключей**, стратегия `consistent_hashing`.
- V = 1, 3, 50, 150
- измерить: `stddev/mean` распределения ключей по узлам

Итого: 4 прогона.

**Всего минимум: 13 прогонов.**

### 8.3. Требования к `report.md`

`report.md` должен содержать:
- таблицу всех прогонов;
- минимум **4 графика**:
  1. Throughput по стратегиям (write-heavy и read-heavy).
  2. Процент мигрированных ключей (И при удалении ноды, и при добавлении): `hash_mod_n` vs `consistent_hashing`. 
     - Необходимо сделать несколько прогонов и на основе их взять среднее/медиану.

- объяснение результатов:
  - почему hash-mod-N создаёт катастрофическую ребалансировку;
  - почему consistent hashing перемещает ≈1/N данных;
  - как vnodes влияют на равномерность и почему;
  - в каких сценариях range-based лучше (range-запросы, data locality).
